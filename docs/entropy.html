Entropy encoders

With the quantized symbols, there is still a lot of redundancy in the data. Entropy encoders further reduce the size of data by assigning short patterns to long strings of repeated symbols.

Goal

When we get our quantized symbols, we can use entropy encoders to further reduce the size of the data. Entropy encoders work by assigning short patterns to long strings of repeated symbols in our image data. This is the last compression stage: it should be lossless with respect to the quantized symbols.

To further explain entropy encoders, we must first define what entropy is. Entropy is a measure of the unpredictability of a source of information. The more predictable the source of information, the lower the entropy. The more unpredictable the source of information, the higher the entropy. Entropy encoders attempt to reach the lower bound of UMich Alum Claude Shannon's sourcing code theorem, which states that you cannot compress data into fewer bits than its entropy without permanently losing information. Shannon's theorem is given by the equation: \[ L \geq H(X) \] where \(L\) is the average codeword length of a compressed symbol and \(H(X)\) is the entropy of the source of information. The entropy of the source of information \(H(X)\) is given by the equation: \[ H(X) = -\sum_{x \in X} p(x) \log_2 p(x) \] where \(X\) is the source of information and \(p(x)\) is the probability of the symbol \(x\) occurring.

In this project, we experimented with Huffman encoding, arithmetic encoding, ANS encoding, and run-length encoding. We found through experimentation that Huffman encoding was the most efficient and reliable entropy encoder for our image data. Our final hybrid SVD-DCT pipeline, thus, uses Huffman encoding for our image data. These following sections here focus on how to encode but to decode, you reverse the process of encoding.

Run Length Encoding

Run length encoding (RLE) is a simple entropy encoder that replaces repeated runs of symbols with a single symbol and a count. A run is a sequence of consecutive symbols that are the same. Given a list of symbols like "green green green red red red red", the RLE encoding would be: (green, 3), (red, 3). Historically, it has been used since 1967 for analog television.

In our project, RLE is used to compress runs of data before they are fed to a more complex entropy encoder. We implemented it using our own Python implementation becuase it was easy to implement ourselves.

Huffman Encoding

Huffman encoding is a variable-length encoding scheme that assigns shorter bit patterns to more frequent symbols and longer patterns to rare symbols. This is achieved by building a Huffman tree from the symbol frequencies. The Huffman tree is a binary tree where each leaf node is a symbol and each edge is a bit.

Building the Tree

The tree is built using the symbols and their frequencies. The symbols with the minimum frequencies are create a new node in the tree that is the sum of the two minimum frequency symbols. This process is repeated until all symbols are in the tree. The root of the tree is the symbol with the highest frequency. Bits are assigned based on the path from the root to the leaf node. The left edge is 0 and the right edge is 1.

Looking at the Huffman encoding of the symbols "AABCCCCDD", we can see that the the symbols and frequencies are: A: 2, B: 1, C: 4, D: 2. To build the tree:

  1. Merge B(1) and A(2) to form BA with frequency 3
  2. Merge D(2) and BA(3) to form DB with frequency 5
  3. Merge C(4) and DB(5) to form CDBA with frequency 9

Traversing the tree from the root to the leaf nodes where 0 is the left edge and 1 is the right edge, we get the following encodings:

  • C: 0
  • D: 10
  • B: 110
  • A: 111

The Huffman encoding of the symbols "AABCCCCDD" would be: 11111111000001010.

Ol' Reliable

In our project, we found Huffman encoding to be the best at compressing our image data reliably. So when we made our final hybrid SVD-DCT pipeline, we decided to use Huffman encoding for our image data. For our implementation, we used the dahuffman Python library as our Huffman encoder. The library would take in a dictionary of symbols and their frequencies and return the raw bytes of the compressed data. We would have to save the dictionary of symbols and their frequencies in the metadata of our compressed image object.

Something surprising was that the ANS encoding was not as effective as the Huffman encoding. What we suspect the problem with ANS encodng was the metadata overhead needed to store components of the ANS state. We used a library simple_ans to implement ANS encoding. The library would take in a dictionary of symbols but would return an EncodedSignal object. That may have been big compared to just raw bytes of the compressed data from the Huffman encoder.

Arithmetic Encoding

Aithmetic encoding is another entropy encoding method that stores entire messages as a single floating point number in the range [0, 1.0). The message is encoded by dividing the range [0, 1.0) into sub-ranges for each symbol in the message. The sub-ranges are determined by the probability of the symbol. So a symbol with a higher probability will have a larger sub-range. The sub-ranges are then used to encode a message.

Encoding a Message

To encode a message, with the sub-ranges determined, an encoder would take the first symbol of the message and find the corresponding sub-range. The encoder would then update the range to the corresponding sub-range. This process is repeated for each symbol in the message. The final range is the encoded message.

Take for example the message "ABC" with the frequencies A: 1, B: 1, C: 1. The sub-ranges would be:

  • A: [0, 0.33)
  • B: [0.33, 0.66)
  • C: [0.66, 0.99)

For the first symbol "A", the sub-range is [0, 0.33). The encoder would then update the range to [0, 0.33). The same probability distribution would be used for the range [0, 0.33) for the next symbol B. The sub-range is [0, 0.1089). This continutes until the last symbol C is reached. The final range of the encoded message would be [5/27, 2/9) ~ [0.185, 0.222)

Floating Point Precision Issues

For really long messages (like image data), the encoder runs into precisions issues with floating point numbers. So the encoder was not the best choice for our compression pipelines and we decided to not use it beyond a single dct pipeline.

Asymmetric Numeral Systems (ANS)

Asymmetric Numeral Systems (ANS), invented by Jarek Duda in 2006, is a modern entropy coding technique that bridges the gap between Huffman coding and Arithmetic coding. It achieves the exact mathematical efficiency of Arithmetic coding (fractional bits) but operates with the sheer speed of Huffman coding.

The LIFO State Machine

Unlike Huffman, which converts symbols directly into bits using a dictionary, ANS operates on a single, continuously updating natural number called the State (\(x\)). Symbols are mathematically "baked" into this integer.

Crucially, this mathematical baking process operates as a Last-In, First-Out (LIFO) stack. Because the mathematical operations are perfectly symmetrical, whatever symbol is baked into the State last will inherently be the first symbol extracted if you run the exact same math in reverse.

The Encoding Loop

The encoder starts with a baseline State integer and a predefined safe operating range (e.g., between 16 and 31). When a new symbol arrives, the encoder must bake it into the State while keeping the integer from overflowing. This requires two perfectly reversible steps:

  1. Renormalize (Extract Bits): First, the encoder checks if baking the symbol will push the State over the maximum limit. If it will, it chops off the bottom bits of the State (by dividing by 2) and pushes those overflow bits directly into the compressed file. It repeats this division until the State is small enough. (Reversing this process simply requires reading bits from the file and multiplying by 2 to pump the State back up).
  2. Bake the Symbol (Grow State): Now that there is room, the encoder scales the State proportionally to the inverse of the symbol's probability using cumulative frequency tables. The math roughly looks like: \(x_{new} \approx \lfloor x / p \rfloor + \text{offset}\). Highly frequent symbols make the state grow very slowly, effectively costing fractional bits. (Reversing this requires checking the remainder \(x \pmod M\) to identify the symbol, then shrinking the State).

This loop—chopping the State down by spitting out bits, then growing it by baking in a symbol—repeats until the entire image is consumed. The "compressed file" is simply the collection of bits that fell off during renormalization, plus the final State integer saved in the header.

Our Implementation & "Probability Fracturing"

We built an experimental pipeline using the simple_ans Python library to encode our quantized DCT coefficients. Mathematically, the fractional bits of ANS should have easily beaten our Huffman encoder's compression ratio. However, in practice, our final file sizes were substantially worse.

We discovered a critical engineering trade-off: metadata overhead. Because our 16-bit DCT quantization created a massive, fractured alphabet (thousands of unique symbols with erratic probabilities), the ANS state machine required an incredibly precise probability table to initialize. The sheer byte-size required to store this fractured dictionary inside the header of our .uofm container completely erased the fractional-bit savings generated by the algorithm. Because of this architectural overhead, we selected Huffman coding as the robust champion for our final Hybrid pipeline.

Hand-off

Once we are done with the entropy encoding of our signals, we package the data into a .uofm container. The container stores the raw bytes of the compressed data and the metadata needed to decode the data. From here we compute metrics like compression ratio, PSNR, and SSIM to evaluate the performance of our compression pipeline.
