Results

Compression ratio, quality tradeoffs, and what worked best across methods.

Each codec was tested against JPEG over 14 test images from the RGB 16-bit dataset. JPEG's quality is adjusted per image to match our codec's output file size as closely as possible, making this a direct apples-to-apples comparison. The metrics tested are PSNR and SSIM, for both, higher is better. Compression ratios are calculated against the original uncompressed file sizes.

Hybrid HF Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.13 dB / 34.32 dB  (90.7% of JPEG)
  SSIM: 0.7889  / 0.8638     (91.3% of JPEG)
COMPRESSION RATIO:
  Best:    136:1  (fireworks.ppm)
  Worst:    70:1  (leaves_iso_1600.ppm)
  Average: ~100:1

The hybrid HF codec is our best all around performer, with an average compression ratio of ~100:1. It also was had our best average PSNR compared to JPEG and not a bad SSIM either. However, it still underperforms JPEG on both metrics for each image that was tested. The main reason for this is our encoder wastes bits on zeros, after quantization, most DCT blocks are filled with zeros, and our Huffman coder still encodes them as symbols. JPEG handles this cleaner by simply stopping when it hits trailing zeros in a block. Adding this feature would free up bits and directly improve quality.

DCT HF Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.40 dB / 35.24 dB  (89.1% of JPEG)
  SSIM: 0.8181  / 0.8704     (94.0% of JPEG)
COMPRESSION RATIO:
  Best:    122:1  (fireworks.ppm)
  Worst:    57:1  (big_building.ppm)
  Average:  ~78:1

The DCT HF codec averages a ~78:1 compression ratio and was the only codec to beat JPEG on a quality metric, it outperformed JPEG on SSIM on fireworks.ppm (0.9364 vs 0.9060). But it still lost to JPEG on every other test image. This ultimately comes down to quantization, JPEG's quantization tables have been fine tuned over decades. Ours is a bit more uniform, so we end up throwing away some of the wrong information and losing quality we didn't need to.

DCT ANS Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.57 dB / 35.62 dB  (88.6% of JPEG)
  SSIM: 0.8249  / 0.8784     (93.9% of JPEG)
COMPRESSION RATIO:
  Best:     81:1  (fireworks.ppm)
  Worst:    41:1  (big_building.ppm)
  Average:  ~60:1

The DCT ANS codec yielded the largest file sizes of the three, averaging a ~60:1 compression ratio. It performed the worst on PSNR and essentially tied with DCT HF for best SSIM. Like the other two codecs, a lot of the bloat comes from how we handle zeros after quantization, we're still encoding when we don't need to, and especially with ANS that inefficiency adds up more than it does with Huffman.

That said, ANS (Asymmetric Numeral Systems) is typically one of the best entropy coders out there, it's used in modern codecs like Zstandard and AV1. If we had more time, a cleaner ANS implementation combined with better quantization and proper zero handling could realistically get us past JPEG. It's probably the most promising path for improvement.

Where We Fall Short and What Could Fix It

  • Encoding zeros wastes bits. After quantization, most of a DCT block ends up as zeros, especially in the high-frequency corners. Instead of skipping them, our encoder codes them as symbols anyway, which bloats the file. JPEG just stops when it hits trailing zeros in a block, which saves a lot of space. Adding something like that to our pipeline would be a solid improvement.
  • Our quantization matrix is too simple. JPEG's quantization tables are tuned to how humans actually see images — we're less sensitive to high-frequency detail, so JPEG throws more of that away. Ours is more of a blanket approach, so we end up keeping things we didn't need and losing things we did. Better quantization tables would improve quality without touching anything else.
  • The arithmetic encoder has a precision bug and had to be discluded from this page. Our DCT AE codec crashes on every image we tested. The issue is that when there are a lot of unique symbols, the probabilities get so small that the floating point math breaks down. It gave errors during testing, which is unfortunate since arithmetic coding is theoretically more efficient than Huffman. Getting this working properly would be a big step forward and could be the final push we need to pass JPEG.
