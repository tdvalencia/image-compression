Results

Compression ratio, quality tradeoffs, and what worked best across methods.

Each codec was tested against JPEG over 14 test images from the RGB 16-bit dataset. JPEG's quality is adjusted per image to match our codec's output file size as closely as possible, making this a direct apples-to-apples comparison. The metrics tested are PSNR and SSIM, for both, higher is better. Compression ratios are calculated against the original uncompressed file sizes.

Experiment setup

Benchmarks are produced by codec/compare_to_jpeg.py over 14 RGB images in images/rgb16bit/ (16-bit PPM). Each image is loaded and preprocessed with the same load_and_preprocess_image path as the encoders (including crop to a multiple of 8×8). Our codec writes a .uofm container; compression ratio is the original uncompressed PPM size divided by that file size (and we report per-codec best/worst/average ratios across the set). For a rate-matched baseline, we encode the same image with Pillow’s JPEG (standard 8-bit path) and sweep quality Q = 1…100, picking the Q whose output size is closest to our codec’s byte count for that image. PSNR is computed in the benchmark script from MSE on the float RGB images (peak value 1.0). SSIM uses scikit-image (data_range=1.0, win_size=5) on full-resolution float RGB in [0, 1]: the reference is the preprocessed 16-bit image scaled by 1/65535, and our reconstruction is decoded 16-bit scaled the same way; the JPEG reconstruction is 8-bit decoded and scaled by 1/255, cropped if needed so height and width match for the metric.

Hybrid Huffman Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.13 dB / 34.32 dB  (90.7% of JPEG)
  SSIM: 0.7889  / 0.8638     (91.3% of JPEG)
COMPRESSION RATIO:
  Best:    136:1  (fireworks.ppm)
  Worst:    70:1  (leaves_iso_1600.ppm)
  Average: ~100:1

The hybrid Huffman codec is our strongest all-around codec if you care about compression ratio first: about 100:1 on average versus ~78:1 (DCT Huffman) and ~60:1 (DCT ANS). At the matched JPEG bitrates in this benchmark it still trails JPEG on both PSNR and SSIM on every image. Among our three variants, the hybrid path also has the lowest average custom PSNR and SSIM at those points—roughly what we expect when we push rate harder than the DCT-only pipelines. The main limiter we see is wasted bits on zeros: after quantization, many DCT blocks are mostly zeros, but our encoder still represents them symbol-by-symbol where baseline JPEG effectively ends the block when the zig-zag hits trailing zeros. A similar EOB-style shortcut would free bits and help quality.

DCT Huffman Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.40 dB / 35.24 dB  (89.1% of JPEG)
  SSIM: 0.8181  / 0.8704     (94.0% of JPEG)
COMPRESSION RATIO:
  Best:    122:1  (fireworks.ppm)
  Worst:    57:1  (big_building.ppm)
  Average:  ~78:1

The DCT Huffman codec averages a ~78:1 compression ratio and was the only codec to beat JPEG on a quality metric: it outperformed JPEG on SSIM on fireworks.ppm (0.9364 vs 0.9060). But it still lost to JPEG on every other test image. This is largely due to quantization: JPEG's quantization tables have been fine tuned over decades. Ours is a bit more uniform, so we end up throwing away some of the wrong information and losing quality we didn't need to.

DCT ANS Codec

AVERAGES (Custom / JPEG):
  PSNR: 31.57 dB / 35.62 dB  (88.6% of JPEG)
  SSIM: 0.8249  / 0.8784     (93.9% of JPEG)
COMPRESSION RATIO:
  Best:     81:1  (fireworks.ppm)
  Worst:    41:1  (big_building.ppm)
  Average:  ~60:1

The DCT ANS codec yielded the largest file sizes of the three, averaging a ~60:1 compression ratio. It performed the worst on PSNR and essentially tied with DCT Huffman for best SSIM. Like the other two codecs, a lot of the bloat comes from how we handle zeros after quantization—we still encode symbols we could skip. In our tests, simple_ans also adds noticeable metadata and state overhead compared to raw Huffman bytes, so the theoretical ANS rate win did not show up until the rest of the bitstream is tightened—not because ANS is inherently worse than Huffman in principle.

Still, ANS (Asymmetric Numeral Systems) is typically one of the best entropy coders out there, it's used in modern codecs like Zstandard and AV1. If we had more time, a cleaner ANS implementation combined with better quantization and proper zero handling could realistically get us past JPEG. It's probably the most favorable course for improvement.

Where We Fall Short and What Could Fix It

  • Encoding zeros wastes bits. After quantization, most of a DCT block ends up as zeros, especially in the high-frequency corners. Instead of skipping them, our encoder codes them as symbols anyway, which bloats the file. JPEG just stops when it hits trailing zeros in a block, which saves a lot of space. Adding something like that to our pipeline would be a solid improvement.
  • Our quantization matrix is too simple. JPEG's quantization tables are tuned to how humans actually see images — we're less sensitive to high-frequency detail, so JPEG throws more of that away. Ours is more of a blanket approach, so we end up keeping things we didn't need and losing things we did. Better quantization tables would improve quality without touching anything else.
  • The arithmetic encoder has a precision bug and was excluded from this page. Our DCT AE codec crashes on every image we tested. The issue is that when there are a lot of unique symbols, the probabilities get so small that the floating point math breaks down. It gave errors during testing, which is unfortunate since arithmetic coding is theoretically more efficient than Huffman. Getting this working properly would be a big step forward and could be the final push we need to pass JPEG.
